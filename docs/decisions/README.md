# Architecture Decision Records (ADR)

This directory contains the permanent architectural decisions of SchoolOS.

Unlike research notes, design discussions or theoretical documents, Architecture Decision Records (ADRs) capture decisions that have been formally accepted by the project.

Each ADR explains:

- the context that led to the decision,
- the architectural decision itself,
- the rationale behind it,
- its consequences,
- and the alternatives that were considered.

The purpose of ADRs is to preserve not only **what** was decided, but also **why** it was decided.

As SchoolOS evolves, theories may become richer, implementations may change, and technologies may evolve.

The architectural principles recorded in this directory provide a stable foundation that guides those changes while preserving the core philosophy of the platform.

---

## ADR Lifecycle

Each ADR has one of the following statuses:

- **Accepted** — The decision is active and part of the architecture.
- **Superseded** — The decision has been replaced by a newer ADR.
- **Deprecated** — The decision is no longer recommended but remains for historical reference.
- **Proposed** — The decision is under discussion and not yet adopted.

## Guiding Principle

Every significant architectural decision within SchoolOS should be documented as an ADR.

Theories explain ideas.

ADRs preserve decisions.

Implementation follows both.
